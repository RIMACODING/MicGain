## VM Verification Results (2026-06-11)

**Machine:** Windows 11, multiple output devices (Audient USB interface, Steam virtual speakers, onboard Realtek HDA)

### Device state enumeration via registry

Registry path: `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\MMDevices\Audio\Render`
Friendly name property key confirmed: `{a45c254e-df1c-4efd-8020-67d146a850e0},2` (PKEY_Device_FriendlyName)

Devices observed (non-phantom only):

| Name | State | RawState |
|---|---|---|
| Headphones | Unplugged | 8 |
| Altavoces | Active | 1 |
| Loop-back 1/2 | Active | 1 |
| Analogue 1/2 | Active | 1 |
| Altavoces | Unplugged | 8 |

- **Altavoces** = Steam virtual speakers (Sunshine/Steam Streaming Speakers) — two registry entries (phantom + active)
- **Loop-back 1/2 / Analogue 1/2** = Audient USB audio interface virtual endpoints — protected, cannot be disabled via Sound settings
- **Headphones** = onboard Realtek HDA jack — physically unplugged

### Findings

#### "No active output device" edge cases — PARTIALLY VERIFIED

- `Active` (state=1) and `Unplugged` (state=8) are **distinct and reliably readable** from the registry ✅
- `Disabled` (state=2) could **not** be triggered on this VM:
  - Audient virtual endpoints (Loop-back 1/2, Analogue 1/2) are protected and cannot be disabled via Sound settings
  - Steam virtual speakers (Altavoces) also resisted disabling
  - Registry state did not update even after `Restart-Service AudioSrv -Force` and a full machine reboot
- **Conclusion**: `Disabled` state requires a machine with a single standard physical output device (e.g. laptop with only built-in speakers). `NEEDS-VM-VERIFICATION` on a simpler machine.
- **Registry lag confirmed**: registry `DeviceState` does not update until Windows Audio service fully processes the change — `AudioDeviceService` MUST use `IMMDeviceEnumerator.EnumAudioEndpoints(eRender, DEVICE_STATE_ACTIVE)` (live COM API), NOT direct registry reads, to get accurate real-time state.

#### Consent dialog device naming — VERIFIED ✅

- 3 active output devices confirmed on this machine simultaneously
- Friendly names are human-readable via `PKEY_Device_FriendlyName` (`{a45c254e-df1c-4efd-8020-67d146a850e0},2`)
- Multiple active devices confirmed — consent dialog naming test is valid on this machine
- `AudioDeviceService` must call `IMMDeviceEnumerator.GetDefaultAudioEndpoint(eRender, eConsole)` and read its friendly name — confirmed this returns a meaningful string on multi-device machines ✅

### Implementation recommendations for AudioDeviceService

1. Use `IMMDeviceEnumerator.EnumAudioEndpoints(eRender, DEVICE_STATE_ACTIVE)` — not registry reads — for live device state
2. Empty result from the above = "no active output device" (covers both Disabled and Unplugged correctly)
3. Use `IMMDeviceEnumerator.GetDefaultAudioEndpoint(eRender, eConsole)` for the consent dialog device name
4. Phantom/NotPresent entries (high-bit state flags) are automatically excluded by the Core Audio API — no extra filtering needed

### Remaining NEEDS-VM-VERIFICATION

- `Disabled` (state=2) edge case: needs a machine with a disableable physical output device (laptop with built-in speakers only recommended)
