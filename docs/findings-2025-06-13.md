# Findings — 2025-06-13 (T3.1 hardening session)

## Bugs found and fixed

### B1 — Missing asset copy in MicGain.App.csproj (blocker)

**Symptom**: On a machine without Equalizer APO, the consent dialog never appeared.
Instead, the main window showed "Equalizer APO was not found on this machine."

**Root cause**: `RunGuidedInstallAsync` resolves the bundled installer at
`<AppContext.BaseDirectory>/assets/installer/EqualizerAPO-x64-1.4.2.exe`,
but the csproj had no `<Content>` or publish items to copy `assets/installer/`
into the build output or publish directory. The installer binary was simply
never landing next to the exe.

**Fix** (`src/MicGain.App/MicGain.App.csproj`):
```xml
<ItemGroup>
  <Content Include="..\..\assets\installer\**\*"
           CopyToOutputDirectory="PreserveNewest"
           CopyToPublishDirectory="PreserveNewest"
           Link="assets\installer\%(RecursiveDir)%(Filename)%(Extension)" />
</ItemGroup>
```

**Status**: ✅ Fixed. `dotnet publish` now includes `assets/installer/EqualizerAPO-x64-1.4.2.exe`
and `assets/installer/README.md` in `publish/assets/installer/`.

### B2 — Fragile audio service restart (&& short-circuit)

**Symptom**: If `net stop audiosrv` fails (service already stopping, timeout,
unexpected state), `&&` prevents `net start audiosrv` from running. The app
would silently leave the service stopped, reporting `Succeeded` when the service
is actually down.

**Root cause**: `ApoInstallService.OfferAudioServiceRestartAsync` called:
```csharp
RunElevatedAsync("cmd.exe", "/c net stop audiosrv && net start audiosrv", ...)
```
The `&&` short-circuits on non-zero exit codes. Additionally, the command was
invoked via `cmd.exe /c`, making the UAC prompt less descriptive.

**Fix** (`src/MicGain.Core/Services/ApoInstallService.cs`):
- Two separate `RunElevatedAsync` calls: `net.exe stop audiosrv /y`, then
  `net.exe start audiosrv`.
- Calls target `net.exe` directly (not via `cmd.exe /c`), improving UAC
  prompt clarity.
- UAC-cancellation tracked independently for each call.

**Status**: ✅ Fixed. Tests updated accordingly.

### B3 — Duplicate detection causing wrong UI path

**Symptom**: On a machine where `HKLM\SOFTWARE\EqualizerAPO\ConfigPath` exists
but the config directory is absent (e.g., from a prior partial install),
`OnStartup` detects APO as "installed" and routes to `ShowMainWindow`, but
`MainViewModel.LoadAsync` re-runs `Detect()` and gets `NotInstalled`, showing
the static "APO not found" message instead of the install consent dialog.

**Root cause**: Detection ran in two places with potentially different results.
`ApoDetectionService.Detect()` checks both the registry key AND filesystem
sanity. If the registry key exists but the config dir was deleted, the second
detection can flip.

**Fix** (`src/MicGain.App/App.xaml.cs` + `src/MicGain.App/ViewModels/MainViewModel.cs`):
- `OnStartup` stores the `ApoDetectionResult` and passes the pre-detected
  `ConfigPath` directly to `MainViewModel`.
- `MainViewModel` no longer re-runs detection on initial load (constructor
  takes `string configPath`).
- The detection service is kept as an optional parameter for use by the Refresh
  button only.

**Status**: ✅ Fixed. Tests updated accordingly.

### B4 — InstallConsentViewModelTests stale assertions

**Symptom**: Two test failures in `InstallConsentViewModelTests`:
- `AcceptCommand_TransitionsToReadyStub_AndRequestsClose` — expected `Ready`
  but got `Installing`.
- `DialogClosedAfterAccept_KeepsReadyState` — same.

**Root cause**: T2.2 wired the real state machine transitions (Accept → Installing,
then CompleteInstall/FailInstall → Ready/InstallFailed). The tests still reflected
the T2.1-only stub behavior where Accept went directly to Ready.

**Fix** (`tests/MicGain.App.Tests/InstallConsentViewModelTests.cs`):
- Updated assertions to expect `Installing` instead of `Ready` after Accept.

**Status**: ✅ Fixed.
